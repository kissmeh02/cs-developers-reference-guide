Security & Zero Trust
Assume breach: verify explicitly, least privilege, and defense in depth — from identity to workload to data at rest.
Zero Trust Principles
Never trust the network
mTLS between services. No "trusted internal VLAN" — lateral movement is the attacker's goal after one phished laptop.
Verify every request
OIDC/OAuth2 for users, SPIFFE/SPIRE or cloud IAM for workloads. Short-lived tokens. RBAC + ABAC where needed.
OWASP Top Risks (web)
RiskMitigation
Injection (SQL, command)Parameterized queries; never concatenate user input into shell
Broken access controlAuthorize every route — IDOR checks (user owns resource)
SSRFBlock internal IPs from user-supplied URLs; allowlists
XSSContext-aware escaping; CSP headers; avoid dangerouslySetInnerHTML
CSRFSameSite cookies + anti-CSRF tokens for state-changing requests
Dependency & Supply Chain

Lockfiles, SBOM (CycloneDX), signed images (cosign), Dependabot/Renovate, and review before merging major upgrades. SLSA levels describe build integrity.

// AEAD

Authenticated encryption (GCM, ChaPoly) gives confidentiality + integrity in one API — prevents bit-flipping attacks.

TLS

TLS 1.2+ only; prefer TLS 1.3. Certificate pinning only when you control clients. HSTS after HTTPS is stable. Never mixed content.

// 18 · security & ai
AI Collaboration
How to pair with LLMs safely: context windows, prompt hygiene, verification, and keeping secrets out of prompts.
Effective Practices
Ground truth
Paste relevant files, error messages, and constraints. Vague prompts get plausible garbage.
Verify everything
Run tests, read diffs, check security implications. Models optimize for plausibility, not correctness.
No secrets in prompts
Treat vendor APIs as untrusted for retention. Use env vars locally; redact in examples.
Iterative refinement
Small steps: generate interface → implement → test → fix. Better than one-shot mega-prompts.
Risks
  • ·Hallucinated APIs — always check docs for functions that don't exist.
  • ·License — generated code may resemble training data; review for compliance.
  • ·Prompt injection — untrusted content in RAG or user messages can override instructions.

Frontend Systems
Browser rendering pipeline, critical rendering path, hydration, and performance budgets for real users on slow devices.
Rendering Pipeline (simplified)
HTML→DOM→CSSOM→Render tree→Layout→Paint→Composite

JS can block parsing (unless defer/async). Large layout thrashing = read/write/read DOM in a loop — batch reads then writes.

Web Vitals (Core)
LCPLargest Contentful Paint — loading performance. Target <2.5s.
INPInteraction to Next Paint — responsiveness. Target <200ms.
CLSCumulative Layout Shift — visual stability. Target <0.1.
SPA / SSR / SSG
CSR
Client renders — fast navigations after load, poor first paint without care.
SSR
Server sends HTML — good SEO and FCP; cost per request.
SSG
Pre-render at build — best for content; rebuild on change.
Hydration
Attach event listeners to server HTML — mismatch = bugs and double work; partial hydration reduces cost.

Type Systems
Static vs dynamic
Static: errors before run (with soundness caveats). Dynamic: flexibility, runtime type tags.
Structural vs nominal
TypeScript: structural — shape matters. Java: nominal — name matters.
Variance
Covariance/contravariance in generics — function params often contravariant; returns covariant.

Compilers
Lexing, parsing, IR, optimization, and codegen — the pipeline from source text to machine code.
Phases
Lexer→Parser→AST→Semantic analysis→IR→Optimize→Codegen

Lexer: regex → tokens. Parser: CFG → AST (LL, LR, PEG). SSA form enables many optimizations (constant folding, dead code elimination).

JIT vs AOT
AOT
Compile before run — predictable startup, native binaries (Go, Rust).
JIT
Profile-guided optimization at runtime — JVM, V8. Warmup latency, peak performance.

// 24 · reference
Golden Rules
Short principles worth internalizing — the checklist before you ship, refactor, or declare victory.
Ship
01
Make it work, make it right, make it fast — in that order.
Optimization without correctness is waste.
02
The database is the hardest thing to change — design schemas with migrations in mind.
Backward-compatible deploys require expand-contract patterns.
03
Observability is not optional — if you cannot measure it, you cannot fix it at 3am.
Metrics, logs, traces — pick a stack and instrument consistently.
Safety
04
Never trust input — validate at boundaries, encode on output.
XSS, injection, and SSRF all start with trusting the wrong bytes.
05
Least privilege everywhere — humans, services, and CI tokens.
Blast radius limits when credentials leak.
06
Secrets belong in vaults and env at runtime — never in repos or images.
Git history is forever.
Collaboration
07
Small PRs, clear commits, tests that fail for the right reason.
Review quality drops with diff size.
08
Document decisions (ADR) — future you will not remember the tradeoff.
Context beats heroic recall.
