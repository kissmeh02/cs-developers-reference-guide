// START MONOLITH — EXTRACT LATER

Microservices are an optimization for organizational scale and independent deployment needs — not a starting architecture. A modular monolith with clean boundaries is easier to extract into services later than a big ball of mud is to clean up. Premature distribution is the root of accidental complexity.

SOLID Principles
S — Single Responsibility
One reason to change
A module has exactly one actor who can require it to change. A UserService that also sends emails violates SRP — the marketing team and auth team both own it now.
O — Open/Closed
Open for extension, closed for modification
Add behavior through new code, not by editing existing code. Strategy pattern, plugin systems. Adding a new payment method shouldn't require editing the checkout flow.
L — Liskov Substitution
Subtypes must be substitutable
If Square extends Rectangle but breaks when you set width/height independently, you've violated LSP. Prefer composition. If you can't substitute without checking the type, the hierarchy is wrong.
I — Interface Segregation
No client depends on unused methods
Split fat interfaces. A ReadableStore and WritableStore beat one Store interface with 10 methods. Classes only implement what they use.
D — Dependency Inversion
Depend on abstractions
Business logic must not import the database driver directly. It depends on a repository interface. The concrete implementation is injected. This is what makes unit testing possible.
DDD — Domain-Driven Design
Bounded Context
A semantic boundary where the domain model has a specific meaning. "Order" in shipping ≠ "Order" in billing. Don't share models across contexts.
Aggregate
Cluster of entities with one root (Aggregate Root). Consistency is enforced within the aggregate boundary. Cross-aggregate = eventual consistency only.
Domain Event
Something significant that happened in the domain: OrderPlaced, PaymentFailed. Past tense. Immutable fact. The basis of event sourcing.
Repository
Collection-like interface for persistence. Domain code never touches SQL. Inject the interface, not the ORM. Testable without a database.
Value Object
Immutable, identified by its values not identity. Money(100, USD) == Money(100, USD). No ID. Examples: Email, Money, Address, DateRange.
Context Map
Documents relationships between bounded contexts. Patterns: Shared Kernel, Customer-Supplier, Anti-Corruption Layer (protect your model from upstream chaos).

// THE KEY RULE

The domain knows nothing about frameworks, databases, or HTTP. It only sees the port interfaces it defines. Everything external adapts to it — not the other way around. This makes the domain testable in pure unit tests with no infrastructure.

CQRS & Event Sourcing
CQRS (Command Query Responsibility Segregation)
Separate write model (commands → events) from read model (optimized query projections). Commands change state. Queries return data. Different models, possibly different databases.
Event Sourcing
Store a log of domain events as the source of truth, not current state. Current state = replay of events. Free audit log. Time travel. Snapshots for performance. Kafka as the event log.

ACID vs BASE
ACID — Relational DBs
Atomicity: all-or-nothing — the whole transaction commits or the whole thing rolls back.
Consistency: DB always transitions between valid states per all constraints.
Isolation: concurrent transactions don't see each other's in-progress work.
Durability: committed writes survive crashes (WAL + fsync).
BASE — Distributed NoSQL
Basically Available: system always responds (may return stale data).
Soft state: state may change even without input (replication propagating).
Eventually consistent: all replicas will converge... eventually. Trades consistency for availability and partition tolerance.

Indexing Strategy
when and how to indexPostgreSQL
-- ALWAYS index foreign keys and common WHERE/JOIN columns
CREATE INDEX ON orders(user_id);          -- FK — prevents table scan on join
CREATE INDEX ON users(email);             -- unique lookup
CREATE INDEX ON events(created_at);       -- time-range queries
 
-- Partial index — smaller, faster for subset queries
CREATE INDEX ON orders(status) WHERE status = 'pending';
 
-- Composite index — column ORDER matters for selectivity and range scans
CREATE INDEX ON orders(user_id, created_at);
-- ↑ supports: WHERE user_id=? and WHERE user_id=? AND created_at>?
-- ✗ does NOT support: WHERE created_at>? alone
 
-- Covering index — include extra columns to avoid heap fetch
CREATE INDEX ON orders(user_id) INCLUDE (total, status);
 
-- Diagnose slow queries:
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT ...;
// THE N+1 PROBLEM

Fetching a list of N users then making N separate queries to fetch each user's orders = N+1 queries. Fix: JOIN, subquery, or ORM eager loading (include/preload). Detectable with query logging. Always log slow queries in dev.

Safe Migration Strategy (Expand-Contract)
1. Expand: add new column (nullable)
→
2. Deploy app that writes to both old + new
→
3. Backfill existing rows
→
4. Deploy app that reads from new column
→
5. Contract: drop old column
// NEVER run migration and app deploy simultaneously

Migration runs first. New code deploys after. Old code must work with new schema (backward compatible). Never add NOT NULL without a DEFAULT. Never rename a column in one step — add + copy + drop.

// 401 vs 403 — most common confusion

401 = "Who are you?" — not authenticated, no/invalid token. 403 = "I know who you are, you just can't do this" — authenticated but not authorized. These are not interchangeable.

API Versioning Strategies
URL versioning
/v1/users → /v2/users. Simple, explicit, cacheable. Most common for public APIs. Version in URL = easy to test in browser.
Header versioning
Accept: application/vnd.api+json;version=2. Keeps URLs clean. Harder to test, not cacheable without Vary header.
Sunset headers
Announce deprecation: Sunset: Sat, 01 Jan 2027 00:00:00 GMT and Deprecation: true. Give consumers 6–12 months notice.

Distributed Systems
The 8 fallacies, CAP theorem, consistency models, consensus (Raft), partitioning, replication, and the design patterns that make distributed systems survivable.
The 8 Fallacies of Distributed Computing
The network is reliable
Packets drop. Links fail. Connections time out mid-request. Plan for it.
Latency is zero
Same-DC: ~0.5ms. Cross-region: ~100ms. Chatty microservices kill you.
Bandwidth is infinite
Serialize efficiently. Protobuf >> JSON. Don't send full objects when you need one field.
The network is secure
Internal traffic must use mTLS too. Zero Trust. Anything on the wire can be read or modified.
Topology doesn't change
Nodes die, IPs change, services autoscale. Use service discovery, not hardcoded IPs.
There is one admin
Multiple teams own different parts. Agree on contracts (schema, SLA). No shared mutable state.
Transport cost is zero
Serialization, connection setup, TLS handshakes, retries all cost real time and money.
The network is homogeneous
Different services run on different hardware, OSes, and library versions.
CAP Theorem
CAP: Choose 2 of 3 during a partition
Consistency: every read returns the latest write (or an error).
Availability: every request gets a response (no error, possibly stale).
Partition tolerance: system works despite network partition.

P is not optional — networks do partition. Real choice: CP or AP.
Real-world examples
CP (prefer consistency): PostgreSQL, HBase, Zookeeper, etcd, CockroachDB
AP (prefer availability): Cassandra, CouchDB, DynamoDB (eventual), DNS

PACELC extends CAP: even without partition, there's a Latency vs Consistency tradeoff on every write.
Consistency Models (weakest → strongest)
EventualAll replicas converge given no new writes. Reads may be stale. DNS updates, shopping carts, social media likes.
Read-your-writesYou always see your own writes. Other clients may not yet. Common safe default for user-facing apps.
Monotonic readsOnce you read a value, you never read an older value. No time travel reads.
CausalCausally related operations seen in order by all. A reply always appears after the original comment.
SequentialAll processes see operations in the same order. Not necessarily real-time order.
LinearizableStrongest. Every operation appears instantaneous. Very expensive. Used by etcd, Zookeeper for distributed locks.
Raft Consensus (used by etcd, CockroachDB, TiKV)

Three states: Follower, Candidate, Leader. Leader election: follower times out → becomes candidate → sends RequestVote RPCs → gets majority → becomes leader. Log replication: leader appends entry → sends AppendEntries to followers → commits when majority acknowledge → applies to state machine. Quorum = ⌊n/2⌋ + 1. Always run an odd number of nodes (3 tolerates 1 failure, 5 tolerates 2 failures).

Partitioning & Replication
Range Partitioning
Rows split by key range (A–M, N–Z or date ranges). Simple. Enables range queries. Hotspot risk if data is skewed — all writes go to the latest partition.
Hash Partitioning
hash(key) % N = partition. Uniform distribution. Kills range query ability. Consistent hashing minimizes reshuffling when adding/removing nodes (used by Cassandra, DynamoDB, CDNs).
Leader-Follower Replication
Leader handles writes, replicates to followers. Followers serve reads (possibly stale). Simple. Leader is a SPOF — needs failover automation (Patroni for Postgres).
Leaderless Replication
Any node accepts writes (quorum write W + quorum read R where W+R > N). Dynamo-style. Cassandra. High availability. Conflict resolution needed (LWW or vector clocks).
Distributed Design Patterns
Circuit Breaker
Track failure rate. Open = fast-fail immediately (don't hammer a failing service). Half-open = probe if recovered. Closed = normal operation. Resilience4j, go-breaker.
Saga Pattern
Distributed transactions without 2PC. Sequence of local transactions, each publishing an event. Compensating transactions on failure. Choreography (events) vs Orchestration (central coordinator).
Outbox Pattern
Write to DB + outbox table in same transaction. Separate poller reads outbox and publishes to message broker. Guarantees exactly-once publishing even if app crashes.

Concurrency & Parallelism
Threads, mutexes, deadlocks, the actor model, async/await internals, Go concurrency patterns, and why data races are the most insidious class of bugs.
Concurrency vs Parallelism
Concurrency
Multiple tasks making progress — possibly by interleaving on one core. About structure and design. You can have concurrency without parallelism (single-core async). Go goroutines, async/await, event loops.
Parallelism
Multiple tasks executing simultaneously on multiple CPU cores. About physical execution. Requires multiple cores. Data parallelism (SIMD) is parallelism without concurrency.
// ROB PIKE: "Concurrency is about dealing with lots of things at once. Parallelism is about doing lots of things at once."

Node.js is highly concurrent but (mostly) not parallel. Go is both concurrent AND parallel — goroutines are multiplexed onto GOMAXPROCS OS threads.

Deadlock — The Four Coffman Conditions
1. Mutual Exclusion
At least one resource held in non-sharable mode.
2. Hold and Wait
Process holding a resource is waiting to acquire more.
3. No Preemption
Resources can't be forcibly taken — only voluntarily released.
4. Circular Wait
A → waits for B → waits for A. Break by imposing a global lock ordering.
// PREVENTION: always acquire locks in the same global order

If every code path acquires mutex_A before mutex_B, circular wait is impossible. Use go test -race and go vet to detect data races. ThreadSanitizer (TSan) for C++. These catch entire classes of concurrency bugs before prod.

Async/Await Internals (JavaScript)
How it works
async/await is syntactic sugar over Promises over the event loop. An await suspends the current async function and yields back to the event loop. The microtask queue picks it up when the Promise resolves. No new thread is created.
Common pitfall: sequential awaits
await a(); await b(); runs serially — total time = a + b. Use await Promise.all([a(), b()]) for parallel — total time = max(a, b). One of the most common performance bugs in Node.js code.
