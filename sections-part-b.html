When to Use Which
Array
Default choice. Sequential access, cache-friendly, O(1) index. Bad for mid-array insert. Use when size is known or growth is appended.
Hash Map
O(1) lookup by key. Frequency counting, deduplication, memoization. Bad for range queries, ordered iteration, or tight memory.
Stack
LIFO. Undo/redo, call stack simulation, bracket matching, DFS iteratively, expression evaluation (shunting-yard).
Queue / Deque
FIFO. BFS, task queues, sliding window maximum (monotonic deque — extremely useful for interview problems).
Heap (Priority Queue)
Always need min or max fast. K largest elements, Dijkstra, event scheduling. Min-heap: parent ≤ children.
Trie
Prefix search, autocomplete, spell checking, IP routing. O(k) ops where k = key length. Faster than hash for prefix operations.
Balanced BST
Ordered data with O(log n) everything. Range queries, floor/ceiling. Java TreeMap, C++ std::map, Python sortedcontainers.
Union-Find (DSU)
Connected components, cycle detection, Kruskal MST. Path compression + union by rank = nearly O(1) amortized. Elegant for graph problems.
Segment Tree
Range queries + range updates in O(log n). Sum, min, max over any interval. More complex than BIT but more powerful.
Fenwick Tree (BIT)
Range sum queries + point updates. Simpler than Segment Tree for these specific operations. O(log n), minimal code.
Hash Table Internals
Open Addressing
Collision → probe next slot (linear, quadratic, double hashing). Better cache locality, no extra allocation. Load factor must stay <0.7. Python dict uses this.
Separate Chaining
Collision → linked list (or tree) at slot. Java HashMap — switches to Red-Black tree at 8+ entries per bucket. Load factor can exceed 1.0.
// HASH FUNCTION QUALITY

Bad hash = clustering. Good choices: MurmurHash3 (fast, non-cryptographic), xxHash (fastest), SipHash (DoS-resistant, used by Rust HashMap and Python 3.3+). Never use identity hash for user input — enables hash-flooding DoS attacks where an attacker crafts keys that all collide.

Array index, hash lookup, stack push. Constant — doesn't grow with input size.

Binary search, balanced BST. Double input = +1 step. Extremely fast for large n.

Graph Algorithm Decision Tree
BFS (Breadth-First)
Queue. Shortest path in unweighted graph. Level-order traversal. Finds shortest path but uses more memory than DFS. O(V+E).
DFS (Depth-First)
Stack/recursion. Cycle detection, topological sort, connected components, maze solving. Less memory than BFS. O(V+E).
Dijkstra
Min-heap + relaxation. Shortest path, non-negative weights. O((V+E) log V). Use when edge weights are positive and you need exact shortest path.
Bellman-Ford
Shortest path with negative weights. Detects negative cycles (run V-1 iterations, then 1 more — if still improving, negative cycle exists). O(VE).
Floyd-Warshall
All-pairs shortest path. O(V³). Small dense graphs. DP: dp[i][j][k] = shortest i→j using only nodes 0..k.
Topological Sort
DFS post-order or Kahn's algorithm (BFS + in-degree). DAGs only. Dependency resolution, build systems, task scheduling.
Kruskal's MST
Sort edges by weight + DSU. Add edge if it doesn't form a cycle. O(E log E). Good for sparse graphs.
Prim's MST
Greedy: start from any node, always add cheapest edge to unvisited node. Min-heap + visited set. O(E log V). Good for dense graphs.
Dynamic Programming Framework
DP — 4-step frameworkTypeScript
// 1. Define state: what does dp[i] represent?
// 2. Find recurrence: dp[i] = f(dp[i-1], dp[i-2], ...)
// 3. Base cases: dp[0], dp[1]
// 4. Direction: top-down (memo) or bottom-up (tabulation)
 
// TOP-DOWN (memoization) — natural recursion + cache
function fib(n: number, memo = new Map()) {
  if (n <= 1) return n
  if (memo.has(n)) return memo.get(n)!
  const result = fib(n-1, memo) + fib(n-2, memo)
  memo.set(n, result)
  return result
}
 
// BOTTOM-UP (tabulation) — faster, no stack, space-optimizable
function fib(n: number): number {
  if (n <= 1) return n
  let [prev, curr] = [0, 1]
  for (let i = 2; i <= n; i++) [prev, curr] = [curr, prev + curr]
  return curr  // O(1) space — rolling array optimization
}

Two Pointers / Sliding Window / Binary Search
core patternsTypeScript
// TWO POINTERS — sorted array, pair sum, palindrome check
let l = 0, r = arr.length - 1
while (l < r) { /* shrink from both ends */ }
 
// SLIDING WINDOW — subarray/substring with constraint
let l = 0, windowSum = 0
for (let r = 0; r < n; r++) {
  windowSum += arr[r]
  while (windowSum > target) windowSum -= arr[l++]
  maxLen = Math.max(maxLen, r - l + 1)
}
 
// BINARY SEARCH — sorted array or monotonic answer space
let lo = 0, hi = n - 1
while (lo <= hi) {
  const mid = lo + ((hi - lo) >> 1)  // safe midpoint — avoid overflow
  if (check(mid)) hi = mid - 1       // or lo = mid + 1 for right bound
  else lo = mid + 1
}

CPU Scheduling Algorithms
FCFS — First Come First Served
Simple queue. No starvation. Convoy effect: one long job blocks all short jobs behind it. Bad for interactive workloads.
SJF / SRTF
Shortest Job First / Shortest Remaining Time First. Theoretically optimal average wait time. Problem: how do you know burst time in advance? Starvation of long jobs.
Round Robin
Time quantum (10–100ms) per process. Fair, responsive, good for interactive. Quantum too small = context switch overhead. Too large = degrades to FCFS.
Priority + Aging
Higher priority runs first. Aging prevents starvation: wait time increases priority. Linux uses CFS (Completely Fair Scheduler) — tracks vruntime, picks lowest.
Multilevel Feedback Queue
Multiple queues with different priorities. New jobs start high-priority. If they use full quantum, drop to lower queue. Interactive jobs stay high. CPU-bound drop down.
Linux CFS
Completely Fair Scheduler (since 2.6.23). Red-black tree ordered by virtual runtime. Always runs process with lowest vruntime. Niceness adjusts vruntime increment rate.
Virtual Memory & Paging
Pages & Page Tables
Virtual address space divided into 4KB pages. OS maintains a page table mapping virtual page number → physical frame. Page fault = page not in RAM → OS loads from swap/disk → resumes process.
TLB (Translation Lookaside Buffer)
Hardware cache for page table lookups. TLB hit = fast (<1ns). TLB miss = page table walk = slow. Context switch flushes TLB — one reason process switches cost more than thread switches.

File Systems
Inodes
Every Unix file has an inode storing: permissions, owner/group, timestamps, file size, and pointers to data blocks. NOT the filename. Directory = mapping of names → inode numbers. Hard link = second directory entry pointing to same inode. Symlink = new inode pointing to a path.
Journaling
Ext4, NTFS, APFS write operations to a journal before applying to the main filesystem. On crash, replay or discard the journal. Prevents corruption. Write-ahead logging (WAL) in databases (Postgres, SQLite) uses the same idea.
VFS (Virtual File System)
Linux abstraction layer — same syscalls (open, read, write) work on ext4, tmpfs, procfs, sysfs. Everything in Linux is a file. /proc contains process info as files. /sys exposes kernel internals.
Copy-on-Write (CoW)
BTRFS, ZFS, APFS, Docker layers use CoW. When you write a page, copy it first. Original stays intact. Enables snapshots, efficient forks, and Docker image layers with near-zero copy cost.

HTTP Evolution
HTTP/1.1 (1997)
Keep-alive, pipelining
6 parallel connections/domain. Head-of-line blocking per connection. Still the baseline for most traffic today.
HTTP/2 (2015)
Multiplexed streams
Many requests over one TCP connection. Binary framing. HPACK header compression. Server push. Still TCP-level HoL blocking.
HTTP/3 / QUIC (2022)
UDP-based transport
QUIC replaces TCP. No HoL blocking per stream. 0-RTT resumption. Built-in TLS 1.3. Default for Google, YouTube, Cloudflare.
WebSocket (RFC 6455)
Full-duplex
HTTP upgrade to persistent TCP connection. Both sides push freely. Real-time chat, live dashboards, collaborative editing, multiplayer.

TLS 1.3 improvements over 1.2
1-RTT (was 2-RTT). 0-RTT resumption. Removed: RSA key exchange, RC4, DES, 3DES, SHA-1, MD5. Forward secrecy mandatory (ECDHE). Encrypted certificates.
0-RTT caveat
Fast but vulnerable to replay attacks. The early data sent in 0-RTT can be replayed by an attacker. Never use 0-RTT for non-idempotent requests (POST, payment endpoints).
